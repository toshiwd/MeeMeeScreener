# app/frontend AGENTS.md

## 責務
- 表示、UI state、入力制御、画面構成、一貫した UI ルールを担う。
- UI state は表示と操作のために使う。業務判定や永続化の都合を持ち込まない。
- データ取得、整形、表示の境界を崩さない。
- UI で直せそうに見えても、原因が backend/domain にあるならそちらを優先する。

## 禁止
- 無言 fallback で見た目だけ整える。
- local fallback や追加 UI state で data contract の歪みを吸収する。
- API 由来の歪みを frontend 側の場当たり分岐で隠す。
- 画面ごとの局所最適を共通 UI ルールより優先する。
- 業務判定、複雑なドメイン判断、データ永続化の責務を持ち込む。

## 変更前
- 症状、原因仮説、変更対象、非対象、影響範囲、回帰リスク、検証手順を整理する。
- どの state が表示用で、どの state が操作用かを分ける。
- パフォーマンス劣化が疑われる変更は影響箇所を先に書く。
- 3 ファイル以上、public interface、backend 連携、shared type 変更は plan 先行。

## 計画が必要な時
- 画面構成の再編。
- UI state の持ち方を変える時。
- API 契約変更に追随する時。
- backend/domain と横断して直す時。
- 見た目だけでなく表示条件や入力制御に触る時。

## 停止条件
- 契約不整合を local fallback で隠すしかなくなった時。
- frontend だけで直すと backend/domain の歪みを固定化する時。
- 同じ見た目崩れを繰り返しパッチしている時。
- 検証で不具合が再現したのに原因が未確定のまま進める時。

## 検証
- 変更後の画面操作を確認する。
- 画面遷移、入力、状態復元、再描画の破綻を確認する。
- 未検証は未検証と明記する。
- 問題が backend/domain 起因なら切り分けて戻す。

## Done
- 画面の責務範囲内で修正している。
- 契約の歪みを UI で隠していない。
- 検証結果と残件を明記した。
