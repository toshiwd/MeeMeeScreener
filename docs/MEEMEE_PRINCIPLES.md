# MeeMee Product Principles

この文書は MeeMee 固有のプロダクト原則をまとめる。作業ルールは `AGENTS.md`、UI の具体実装は `docs/ux/UX_POLICY.md`、詳細仕様や Runbook は `docs/CODEX.md` を参照する。

## 目的

MeeMee は、日本株の監視・候補抽出・建玉管理・売買判定更新を一体化した、個人投資家向けの裁量支援ツールである。最優先は「次に何を見るか」と「今日の引けでどう動くか」を速く正確に判断できることに置く。見た目の派手さより、チャートの可読性、判断導線、再現性を優先する。

## 北極星

- 候補抽出は速く、エントリー判断は深く。
- MeeMee 本体は軽く、重い検証は外付けにする。
- チャート閲覧性を壊してまで情報を増やさない。
- 人間の最終判断を補助し、AI が勝手に断定しすぎない。
- ルールは検証可能な形で保存し、再利用できるようにする。

## プロダクトの基本思想

### 候補抽出と売買判断は分ける

- ランキングは「見る順番」を決めるためのものとする。
- 実際の仕掛け判断は、月足・週足・日足の整合、ローソク足、移動平均線、出来高、建玉状況で行う。
- 「上昇しそう」「下落しそう」と「今日入るべき」は別問題として扱う。

### チャートを主役にする

- 画面設計は常にチャート可読性を最優先にする。
- 情報追加はサイド領域、折りたたみ、別タブ、別画面で吸収する。
- メインの表示面積を奪う機能追加は原則として避ける。

### 終値基準で運用する

- MeeMee の判断は原則として終値確定ベースで扱う。
- 場中のブレを前提にしたロジックは補助扱いに留める。

### 建玉は `売-買` で統一する

- 文章、UI、AI 出力、API 返却では必ず `売-買` 形式を使う。
- 例: `2-6` は「売り2、買い6」を意味する。
- 外部 CSV で `上=買い / 下=売り` の表記でも、MeeMee 内部では `売-買` に正規化する。

### 玉操作は段階的に扱う

- 基本単位は `2 -> 3 -> 5` を想定する。
- 低位株では倍量運用の余地を残す。
- 反対玉ヘッジを前提とし、フル片張りを標準にしない。

## 画面設計の原則

### 通常画面

- 目的は「今見るべき銘柄を選び、チャートを読むこと」に絞る。
- チャート、主要 MA、建玉、最低限のランキング理由を優先する。
- 表示密度は高くてよいが、情報の種類を増やしすぎない。

### 外付け検証

- 外付け検証スタックの正式名称は `Tradex` とする。`Tradex` は Trade + CODEX による開発・検証基盤を意味する。
- `Toredex` は誤記として扱い、今後の文書・設定・ユーザー向け表現では使わない。
- Tradex が生成する重い DB、replay、walk-forward、最適化ログ、学習成果物は `C:` に置かず、既定で `G:\Tradex` 配下へ保存する。
- 推奨レイアウトは `G:\Tradex\runs`、`G:\Tradex\data`、`G:\Tradex\artifacts` とする。

- 目的は「ルールの有効性確認」「過去比較」「学習評価」「AI 支援」とする。
- MeeMee 本体とは分離する。
- Snapshot、1M/3M/6M 比較、類似局面、事後統計、walk-forward、学習系のような重い機能は外付け検証側へ寄せる。
- MeeMee 本体に残すのは、publish 済み結果の表示と、配布ユーザー向けの売買判定更新だけに留める。
- 互換性のため、当面はコード内部の `toredex` 名や既存ディレクトリ名が残ってもよい。ただし新規の正式名称と保存先は `Tradex` / `G:\Tradex` に統一する。

### 建玉表示

- 飯塚式表示と通常の数量表示は分ける。
- `6-1` のような比率表示と、`600 / 100` のような株数表示は別列で扱う。
- 買い単価と売り単価も分離表示する。
- スクロール時に文脈を失わないよう、ヘッダー固定を優先する。

### モバイル

- PC 版の情報をそのまま縮小コピーしない。
- モバイルでは「候補確認 -> チャート確認 -> 建玉確認 -> 今日の方針」の順に絞る。
- 横スクロール依存の表を増やさない。

## チャート分析と AI 出力の標準

### チャート分析の標準フレーム

AI がチャートを扱うときは、原則として次の順で整理する。

1. 月足: 環境認識
2. 週足: トレンド発生 / 継続 / 失速
3. 日足: 今日の足と移動平均線の評価
4. 現在の建玉: `売-買`
5. 今日の引けでの具体アクション: `売-買 -> 売-買`

例:

- `1-5 -> 2-5` ヘッジ強化
- `2-6 -> 2-3` 半分利確
- `3-0 -> 0-0` 撤退

### AI 出力原則

- 結論を先に書く。
- 根拠を短く追える形にする。
- 曖昧な表現を減らす。
- 建玉の変化を具体的に書く。
- 可能なら無効化条件も添える。

悪い例:

- 「様子を見てもいいかもしれません」

良い例:

- 「週足は上昇継続。日足は20MA上の押し目候補。1-5のまま維持。終値で20MA割れなら1-3へ縮小」

## データとランキングの原則

### データソース

- 株価データ
- スクリーニング結果
- 建玉履歴
- 取引履歴 CSV
- 信用需給、イベント情報
- EDINET などの補助情報

### データ設計

- 生データは捨てない。
- 正規化テーブルを分ける。
- 画面表示用の派生テーブルやキャッシュを別管理する。
- 日付、銘柄コード、足種は必ずキーとして追えるようにする。
- 欠損時の扱いを実装前に決める。
- AI に再利用させる情報はマシン可読形式でも残す。
- 通常画面に重い処理を混ぜず、検証用途の重い集計は MeeMee 外部の別ジョブ、事前計算を優先する。

### ランキング設計

- ランキングは候補抽出専用とする。
- 当日、前日、前週、前月など、時間軸別ランキングを持てるようにする。
- 上昇と下落を対称に扱う。
- スコア理由を人間が読める粒度で表示する。
- 根拠不明の複合点数は避け、最低でも寄与要素を出す。

## MeeMee 向け実装チェック

### AI エージェントが毎回見る点

- UI 変更時は、通常のチャート閲覧性を悪化させていないか確認する。
- データ項目追加時は、保存先、更新頻度、欠損時挙動、表示先をセットで定義する。
- 文言は投資判断時に誤読しにくい日本語を使う。
- 実装後は最低限の確認手順を明記する。

### 実装で避けること

- 一度に大改修しない。
- チャート領域を恒常的に狭くしない。
- 意味の曖昧な略語を増やさない。
- 根拠の薄い AI スコアを主役にしない。
- 候補抽出スコアと売買トリガーを混同しない。
- 指標だけ増やして、最終的な判断導線を悪化させない。

### フロントエンド

- 既存の表示導線を壊さない。
- コンポーネントは責務単位で分割する。
- 一時しのぎの state 増殖を避ける。
- URL パラメータで持つべき状態とローカル state を分ける。
- 表示切替は「通常 / 外付け検証」の文脈を意識する。

### バックエンド

- まず再現可能な集計を作る。
- 学習モデルは後付けで乗せる。
- 計算コストの高い処理はキャッシュまたは事前計算する。
- CSV 取り込みは列揺れ、表記揺れ、空データに強くする。

### 命名

- 曖昧な略語を避ける。
- 飯塚式や既存ルールと整合する名称を優先する。
- 画面に出す名称と内部名が大きくズレないようにする。

## 最低限のテスト観点

1. チャートの表示面積が減っていないか
2. ランキング順が意図通りか
3. 建玉表示が `売-買` で統一されているか
4. CSV 列揺れで壊れないか
5. データ 0 件 / 1 件でも落ちないか
6. モバイル幅で崩壊しないか
7. 重い画面遷移で固まらないか
8. 外付け検証機能が MeeMee 本体を遅くしていないか

## 変更提案テンプレート

MeeMee に関する変更提案は、次を短く埋める。

### 目的

この変更で何を改善するか。

### 変更内容

何をどこに追加または修正するか。

### 守ること

チャート可読性、既存導線、速度など、何を壊さないか。

### リスク

誤集計、順位変化、UI 崩れ、遅延など。

### 確認方法

最小限の手動確認手順。

## 禁止事項

- 通常画面に検証用情報や学習用情報を詰め込みすぎる。
- シグナル名だけ増やして定義を残さない。
- 既存 CSV 仕様を無言で変える。
- 建玉表記を画面ごとに変える。
- チャートを見えにくくする固定領域を増やす。
- AI の説明だけ立派で、実装が追えない状態にする。

## 最後に

MeeMee の目的は情報量を増やすこと自体ではない。候補抽出、チャート読解、建玉判断、検証を一貫した導線で回せるようにすることが目的であり、追加機能はすべてこの目的に従属させる。
